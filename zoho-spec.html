SPEC B — Zoho CRM & Zoho Creator Implementation

Consultant-facing specification for implementing a governed, scalable lifecycle system using Zoho.
Audience: Integration-aware, non-technical stakeholders

1. Overview (Plain English)

The purpose of this implementation is to use Zoho CRM and Zoho Creator together to manage a complete onboarding and operational lifecycle — from first contact through to active operation — with clear rules, visibility, and accountability.

Important framing:
Zoho is the orchestration and governance layer.
It decides what can move forward, when, and why.

Document intelligence (OCR, validation, classification) is handled by an external document system. Zoho consumes the results and enforces progression rules.

2. High-Level Architecture

Component Responsibility
Zoho CRM Companies, deals, lifecycle stages, approvals, ownership, and commercial visibility.
Zoho Creator Detailed onboarding workflows, forms, operational data, learner-level records, client portals, learner portals, and bulk learner management (grouped by client).
External Document Platform Document ingestion, OCR, validation, and exception detection.
Zoho CRM + Creator together form the system of record.
The document platform is a specialist subsystem that feeds results into Zoho.

3. Lifecycle Stages & Hard Gating

The consultant must implement hard-gated lifecycle stages. Records may not progress unless specific conditions are met.

Minimum required stages

  1. In Contact
  2. NDA Signed
  3. Documents Submitted
  4. Vetting Approved
  5. SLA Signed
  6. Onboarding (Finance + Ops)
  7. Active
Key expectation:
Stage changes should be rule-driven by default, not manual.

4. Governance & Controls

Zoho must enforce:

  • Mandatory fields per stage
  • Required document signals from the document platform
  • Explicit approvals where exceptions are allowed

Overrides

Overrides must:

  • Be intentional (no silent bypassing)
  • Record who approved
  • Require a reason
  • Be auditable later

5. Forms & Data Capture

Zoho Creator will be used to build progressive, saveable forms for onboarding and operations.

  • Application forms
  • Finance onboarding
  • Operational onboarding

Form expectations

  • Users can save and return later
  • Clear progress indicators
  • Contextual guidance where confusion is likely

6. Document Dependency Model

Zoho does not store documents, but it must understand:

  • Which documents are required
  • For whom (company vs individual)
  • At which stage
  • Whether requirements are satisfied
Think of Zoho as the rulebook:
It knows what is required and blocks progress when requirements are unmet.

7. Integration with the Document Platform

The consultant must design a clean, vendor-agnostic integration contract.

What Zoho sends

  • Entity identifiers (company, person)
  • Required document matrix
  • Lifecycle context

What Zoho receives

  • Document status per person
  • Exception flags
  • Completion signals (“gate satisfied”)
Zoho must remain flexible — the integration should not be hardcoded to a single document vendor unless explicitly agreed.

8. Operational Visibility

The implementation must provide dashboards showing:

  • Applications stuck at each stage
  • Common document failure points
  • SLA breaches
  • Pipeline volume and projections

These views are for operations and management, not just sales.

8.1 Learner management dashboards

The system must support managing large numbers of learners (thousands per client) with high-level status visibility:

  • Client-grouped views: All learners organized by client/company, allowing bulk operations and status rollups at the client level
  • High-level status dashboard: Aggregate views showing:
    • Total learners per client
    • Document completion percentages
    • Exception counts and trends
    • Stage distribution (how many at each lifecycle stage)
  • Individual learner management: Drill-down capability to view and manage documents at the individual learner level, with full document status, exceptions, and history
  • Bulk operations: Ability to perform actions across multiple learners (e.g., bulk document requests, status updates, filtering)

9. Client and Learner Portals

The implementation must provide self-service portals for both clients and learners to reduce manual administrative overhead and improve transparency.

9.1 Client portal

A client-facing portal that allows client administrators to:

  • View high-level status dashboards for their organization showing aggregate learner status, document completion rates, and exception summaries
  • Access learner lists grouped by project/program with filtering and search capabilities
  • View individual learner records and document status
  • Upload documents on behalf of learners (when authorized)
  • Submit bulk learner information and coordinate document collection
  • View exception reports and missing document lists
  • Track progress against SLAs and compliance requirements

9.2 Learner portal

A learner-facing portal that allows individual learners to:

  • View their personal profile and current status
  • See which documents are required, which have been submitted, and which are pending or rejected
  • Upload documents directly through the portal
  • Receive notifications about missing or expiring documents
  • View document verification status and rejection reasons (where applicable)
  • Access their compliance history and status
Design principle:
Portals should reduce the need for manual email/phone coordination while maintaining appropriate access controls and data isolation between clients.

10. Security & Access

  • Role-based access (Sales, Vetting, Finance, Ops)
  • Clear ownership of records
  • No shared accounts
  • Full activity history per record

11. Required Consultant Deliverables

  1. High-level data model (entities and relationships)
  2. Lifecycle and gating diagram
  3. Integration contract definition
  4. Governance rules and override logic
  5. Admin documentation
  6. Handover and training
The goal is not just a working system — it must be understandable, supportable, and governable.
